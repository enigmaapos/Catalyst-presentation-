6. Tokenomics (CATA)

Token Name: Catalyst (CATA). Type: ERC-20 Utility & Governance Token.

Deflationary / Emission Mechanics

  • Deflationary: Harvest burn fee (dynamic %), Unstake burn fee, Collection registration fees, Voluntary burns.
  • Emission: Staking NFTs mints new CATA. Base reward rate dynamically adjusts with participation.

Hard-Coded Supply and Staking Caps (Aligned)

  • CATA token supply is **hard-capped at 1,000,000,000**. Only the Staking contract can mint new supply.
  • Global staking capacity capped at **500,000,000 NFTs**: 375M Term stakes / 125M Permanent stakes.
  • **Per-Collection Cap**: A hard cap of **20,000 staked NFTs** is enforced per registered collection contract to prevent single-collection dominance.

Immutable Fee Distribution (CRITICAL)

90% Burned
9% Treasury
1% Owner (Deployer)

This split applies to ALL protocol fees (registration, unstake, harvest, forfeits) and **cannot be modified by governance**.

Top 1% Burner Bonus

Every cycle (~weekly), the top 1% of wallets ranked by burned CATA receive a bonus from the Treasury, capped (default: 5% of Treasury per cycle).

7. Collection Tiers

Verified (Standard)

Normal registration & staking fees. Full privileges. **Custodial staking** inside the protocol.

Unverified (Junk / Rug Survivors)

Requires registration + **surcharge escrow**. Reduced governance rights until upgraded. Forfeited escrow follows the **90/9/1 split**.

Blue-Chip (Curated / Trusted)

**Non-custodial model**: Wallets register directly with a one-time fee for reward access, avoiding custody risk. Alternatively, they can stake via the Verified path.

8. Governance Model

Burn-Weighted Collection Governance (BWCG)

  • Voting weight scaled by **staking contributions & burn activity**.
  • Per-collection vote cap prevents monopolization.
  • Stake-age requirement prevents flash-stake manipulation.
  • Blue-Chip wallets gain eligibility via one-time fee or staking.

Proposal Types Governed

Governance controls base reward adjustments, fee updates, voting parameter changes, collection tier upgrades, **Guardian Council reseeding**, DAO upgrades, and Council lock clearing.

9. Guardian Council Security System (DRS)

Catalyst secures its upgradeable contracts using the **Batch Guardian Council (BGC)** ‚Äî a two-batch guardian model (Active/Standby) with 7 guardians in each batch.

Recovery Thresholds (7 Guardians)

  • 5 of 7 approvals → Recovery valid.
  • 6 of 7 approvals → Raises a Warning and grants the Last Honest Guardian a **48h veto**.
  • 7 of 7 approvals → Auto-Lock triggered, **Standby batch activated immediately**.

Execution Timelock (CRITICAL ALIGNMENT)

Once a recovery proposal achieves the **5-of-7 threshold**, there is a mandatory, non-waivable **4-hour timelock** (EXECUTION_TIMELOCK) before the proposed action can be executed.

This delay provides a final chance for monitoring and intervention before the proposed change is executed.

13. Technical Appendix ‚Äî Catalyst Smart Contract Architecture (Aligned with Current Contracts)

Catalyst is deployed as a modular, upgradeable protocol using the UUPS proxy standard.

Immutable Rule (Restated)

All fees follow the **90% Burn / 9% Treasury / 1% Deployer** split, enforced at the contract level and **unchangeable** by governance.

Core Contracts (UUPS Upgradeable)

  • CataERC20Upgradeable.sol: Max Supply: 1,000,000,000 CATA (hard cap). Minting restricted to Staking contract.
  • CatalystStakingUpgradeable.sol: Custodial NFT staking. Global Cap: 500M NFTs. Per-Collection Cap: 20K NFTs.
  • CatalystGovernanceUpgradeable.sol: On-chain Burn-Weighted Collection Governance (BWCG). Governed by BGC.
  • BatchGuardianCouncilUpgradeable.sol: Dual-Batch Model. Recovery Threshold: 5-of-7. **Mandatory 4-hour Execution Timelock**.

Access Control Hardening

The generic OpenZeppelin role functions (`grantRole()`, `revokeRole()`, and `renounceRole()`) are **overridden to revert** in all core contracts, ensuring all admin changes are auditable via the governance or recovery track.
